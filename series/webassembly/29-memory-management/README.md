# 29 — Memory Management: Ownership Across the Boundary

> **Type:** Explanation

## Two memory worlds

When Rust and JavaScript interact via Wasm, two garbage collectors (or "garbage strategies") coexist:

| Side | Strategy |
|------|---------|
| JavaScript | Tracing garbage collector — objects freed automatically |
| Rust (Wasm) | Ownership + borrow checker — memory freed deterministically |

`wasm-bindgen` bridges these worlds, but you must understand the rules to avoid leaks or use-after-free bugs.

## How exported structs work

When you export a Rust struct via `#[wasm_bindgen]`, an instance is heap-allocated in Wasm linear memory. JavaScript receives a **handle** (an integer index into an internal table). The JS GC tracks the handle object, but **not** the underlying Rust allocation.

```
JS side:
  let obj = new MyStruct();  // handle object (tracked by JS GC)
  
Wasm side:
  MyStruct { ... }           // heap allocation (NOT tracked by JS GC)
```

When the JS handle is GC'd, the Rust allocation is **not automatically freed** — it leaks.

## The solution: explicit `.free()`

Every exported struct gets a `.free()` method generated by `wasm-bindgen`:

```javascript
const s = new MyStruct(42);
const result = s.compute();
s.free();  // Drops the Rust MyStruct, frees its memory
```

After calling `.free()`, the handle is invalidated. Any further use throws an error.

## The `Drop` trait

Rust's `Drop` trait still works for cleanup — it runs when `.free()` is called:

```rust
#[wasm_bindgen]
pub struct Resource {
    data: Vec<u8>,
}

impl Drop for Resource {
    fn drop(&mut self) {
        web_sys::console::log_1(&"Resource freed!".into());
    }
}
```

## Weak references (experimental)

The `wasm-bindgen` team is working on **finalizer-based garbage collection** using the `FinalizationRegistry` browser API. This would allow automatic cleanup without explicit `.free()`. As of 2024, this is available under the `--weak-refs` flag but is not yet recommended for production.

## Strings and temporary borrows

String slices (`&str`) do **not** require `.free()` — they are copied at the boundary. Returned `String` values are owned by JS and freed by the JS GC (wasm-bindgen allocates them in JS's ArrayBuffer, not Wasm memory).

## Practical rules

1. **Always call `.free()`** on exported structs when done.
2. **Don't store Wasm objects beyond their useful life** — use them and free them.
3. **In frameworks** (Leptos, etc.) — the framework manages this for you.
4. **For closures** — use `Closure::once` for fire-and-forget or store `Closure::wrap` results in a Rust struct that manages the lifecycle.
5. **For large data** — prefer passing `Vec<u8>` / typed arrays rather than keeping a long-lived Wasm struct.

## Memory leak detection

Use Chrome DevTools → Memory → Heap Snapshot, compare snapshots over time. Look for growing `WasmModule` memory allocations if you suspect leaks.
